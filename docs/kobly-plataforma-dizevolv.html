Pendências do cliente — integrações de checkout
Para o sistema gatilhar os e-mails, as APIs das plataformas de pagamento precisam estar ligadas
⚠ 5 integrações▼

O motor de automação só dispara campanhas quando recebe um evento de checkout (compra aprovada, Pix gerado, abandono de carrinho…). Esse evento chega pelo webhook-receiver. Logo, cada plataforma de pagamento abaixo precisa ser conectada — o cliente configura o webhook/postback na conta dela apontando para a URL do sistema e fornece as credenciais/secret. Enquanto isso não acontece, nenhum e-mail é gatilhado por aquela plataforma.

📋 O que cada integração precisa, em resumo

1. Acesso/credenciais da API da plataforma (key, token ou Hottok) — fornecidos pelo cliente.
2. Webhook/postback configurado na conta da plataforma apontando para a URL do webhook-receiver do sistema.
3. Signing secret (quando a plataforma assina o payload) cadastrado na tela de Integrações → Webhooks, para validação HMAC.
4. Um evento real de teste para confirmar o mapeamento dos tipos de evento (compra aprovada, Pix, abandono…) antes de ativar campanhas.

Perfis & RBAC — quatro papéis, um produto multi-tenant
Cada perfil tem navegação, dados e permissões isolados por organização
4 perfis · 5 flags▼

Cliente home: dashboard

Dono da conta (e-commerce). Cria e roda campanhas. Nav: dashboard, campanhas, leads, integrações, chamados, planos, ajuda, perfil. can: editCampaign ✓

Gestor home: dashboard

Agência — visão consolidada de várias contas. Nav: dashboard, clientes, campanhas, relatórios, chamados, planos, ajuda, perfil. can: editCampaign ✓ · viewAllAccounts ✓

Suporte home: chamados

Atende chamados e chat de suporte. Nav: chamados, ajuda, leads, perfil. can: answerTickets ✓ · viewAllAccounts ✓ (leads read-only)

Administrador home: segurança

Plataforma — usuários, planos, segurança, auditoria. Nav: segurança, planos, relatórios, clientes, perfil. can: manageUsers ✓ · createPlan ✓ · viewAllAccounts ✓

As 5 flags de permissão são editCampaign · viewAllAccounts · createPlan · manageUsers · answerTickets. No front, o gate de rota vive no AppShell: se a view atual não está no role.nav, cai na role.home; a NavRail só renderiza os itens permitidos e cada tela checa store.can.* antes de agir. No back, o isolamento entre organizações é garantido por RLS no Postgres (não dá pra burlar pelo cliente). Os 4 perfis vêm do enum tipo_user_geral.

Arquitetura & fluxo de dados
SPA React sobre Supabase — como os dados entram e saem
SPA · RLS-scoped▼

SPA em React 18 + Vite, sem react-router: a "navegação" é uma única string view no store; o AppShell mapeia a view ativa para uma tela e renderiza uma de cada vez. A sessão é Supabase Auth real (fases loading → login → recovery → app) dirigida por onAuthStateChange.

Tela chama KoblyApi→ loadDB() (cache + dedup)→ hydrate(): 16 selects paralelos→ RLS filtra por usuário→ reshape snake→camel aninhado

📥 Leitura

supabaseDb.js faz 16 selects em paralelo e remonta os dados (organizations, profiles, plans, templates, tags, emails, domains+dns, webhooks, leads+tags+metrics, campaigns+stats+flows+steps, webhook_events, suporte, transactions, logs, sessões, faq) num grafo aninhado que casa com o contrato da UI. Tudo limitado pelo RLS do usuário logado.

✍️ Escrita

KoblyApi (em mockApi.js — apesar do nome, é a API viva) grava direto via supabase-js e por RPCs (create_managed_org). Edge Functions cuidam de webhook, worker, e-mail e IA.

🧩 Config estática

mockData.js (KoblyMockDB) ainda fornece config estática — papéis, NAV, mapas de tom de badge e ícones de card — espelhando o modelo Bubble original (27 data types, 15 option sets).

🔌 Cliente

supabaseClient.js: client único, persistSession, autoRefreshToken, detectSessionInUrl (links de recovery) e fluxo PKCE.

5 etapas▼
EtapaComponenteO que faz
01webhook-receiverEdge Function pública (verify_jwt=false) e provider-aware. Dois caminhos: token na URL resolve o webhook (provider vem da linha, nunca do corpo) ou caminho legado generic (secret + tipo_evento no corpo). Lê o corpo cru primeiro para verificar HMAC (quando o adapter exige). Rate-limit de 120/min por org. Traduz o payload nativo via adapter; grava webhook_events (idempotente por webhook_id + id_webhook).

04process-stepsWorker (verify_jwt=false) rodado por pg_cron a cada minuto. Varre até 100 etapas devidas (status Iniciado/Em andamento, run_at ≤ agora): Envio → manda via Resend, grava email_events e incrementa lead_metrics + campaign_stats; Adicionar/Remover Tag → muta lead_tags. Marca Finalizado (idempotente); erro grava last_error + attempts.

Camada de IA — DeepSeek via Edge Function segura
Quatro capacidades; chave sempre no servidor, nunca no browser
4 capacidades▼

Toda IA passa pela Edge Function ai-chat (verify_jwt=true), que faz proxy seguro do DeepSeek (deepseek-chat) com a chave lida do Supabase Vault por service_role. O front (KoblyAI) nunca vê a chave; cada chamada empacota contexto real RLS-scoped.

✉️ Gerar e-mail task=email

De um briefing livre, retorna JSON estruturado (assunto, eyebrow, título, parágrafos, CTA, cupom) renderizado pelo design de e-mail Kobly. Fallback estático se a IA falhar.

💡 Sugestão de campanha task=suggestion

1 recomendação curta a partir das estatísticas/criticidade da campanha. Fallback canônico por faixa de criticidade.

📈 Sugestão do dashboard task=suggestion

Resumo/recomendação consolidada da visão atual (conta ou agência).

💬 Assistente task=chat

Chat multi-turno no FAB global, com contexto de campanhas/contas/leads reais; resposta curta renderizada via markdown→HTML.

Segurança — isolamento, segredos e verificação
RLS multi-tenant, Vault, HMAC e least-privilege
defense-in-depth▼

🔒 RLS multi-tenant

77 políticas em todas as 35 tabelas. O pivô é has_org_access(org): admin ou org ∈ my_org_ids (org própria ∪ memberships). Tabelas-filhas herdam acesso por EXISTS no pai. Nenhuma policy para anon.

🗝️ Segredos no Vault

Chaves de Resend e DeepSeek ficam no Supabase Vault, lidas server-side via RPC get_secret com service_role — nunca no browser.

✅ Verificação de webhook

Assinatura HMAC-SHA256 sobre o corpo cru quando o provider exige (requiresSignature); comparação case-insensitive. Rate-limit de 120/min por org.

🧱 Least-privilege

EXECUTE revogado de public e anon nos helpers SECURITY DEFINER; funções de trigger não são chamáveis por RPC; guards anti-escalonamento em profiles/organizations.

📌 Notas honestas (dívidas a tratar antes de produção)

• get_secret é dependência crítica de 4 Edge Functions mas não está em nenhuma migration — foi criada direto no banco (infra não-versionada).
• A anon key está hardcoded em texto plano dentro de 0015_automation_cron.sql.
• Os usuários de auth demo (senha kobly-demo-2026) e seeds são apenas para demonstração — rotacionar antes de produção.
• Adaptadores hotmart · kiwify · perfectpay · kactus estão no allowlist mas ainda não implementados; resend-admin existe no código mas não está deployada.
• Colunas legadas codigo_sendgrid / id_sendgrid são herança do modelo antigo — o envio real hoje é via Resend.

Resumo do produto

  1. SaaS multi-tenant de automação de e-mail para e-commerce, reconstruído de Bubble + n8n para React 18/Vite + Supabase — motor de automação próprio (sem n8n).
  2. Eventos de checkout chegam por webhook provider-aware (generic + nexopayt), criam leads e disparam campanhas com fluxos visuais de 5 tipos de card.
  3. O fluxo agenda etapas (scheduled_steps); um worker (process-steps via pg_cron, a cada minuto) envia por Resend e rastreia tudo, de forma idempotente.
  4. Estatísticas (abertura, CTR, vendas recuperadas, criticidade por trigger) calculadas automaticamente; IA DeepSeek gera e-mails, sugestões e chat via Edge Function segura.
  5. Segurança por padrão: 77 políticas RLS, 4 perfis com RBAC, segredos no Vault, HMAC nos webhooks — sobre 35 tabelas, 17 enums, 15 funções SQL, 34 triggers e 128 índices.
